# PWA Update Detection — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Detect new frontend versions reliably — auto-reload on a fresh start, sticky toast with manual reload during active sessions.

**Architecture:** Two small frontend changes. (1) Toast component honours `duration: 0` (or any non-positive value) as "do not auto-dismiss". (2) The PWA registration callback splits its `onNeedRefresh` handler by elapsed time since boot: ≤ 5 s = silent auto-reload, > 5 s = sticky toast with action button + idle-reload fallback.

**Tech Stack:** React + TSX, vite-plugin-pwa, existing `useNotificationStore`. No new dependencies.

## Task 2: Boot-window vs mid-session PWA update paths

**Files:**
- Modify: `frontend/src/core/pwa/registerPwa.ts`

- [ ] **Step 1: Add boot tracking, constants, and the split onNeedRefresh handler**

Open `frontend/src/core/pwa/registerPwa.ts`. Replace the entire current `registerPwa` function (lines 53-85) with the version below. The `reloadWhenIdle` helper above it (lines 6-51) stays unchanged.

```typescript
const BOOT_GRACE_MS = 5_000

/**
 * Registers the service worker generated by vite-plugin-pwa.
 *
 * Only runs in production builds — the dev server intentionally has no
 * service worker so Vite HMR keeps working and stale caches don't confuse
 * local development.
 *
 * On update we split by how long the app has been running:
 *
 * - Within the boot grace window (≤ 5 s since registerPwa was called):
 *   the user just opened the app. Activate the waiting SW and reload
 *   silently — no toast.
 * - After the boot grace window: show a sticky "new version available"
 *   toast with a "Jetzt neu laden" action button. Also kick the
 *   reloadWhenIdle path as a background fallback so users who ignore
 *   the toast still pick up the update on their next idle moment.
 */
export function registerPwa(): void {
  initInstallPrompt()

  if (!import.meta.env.PROD) return

  const bootedAt = performance.now()

  void import("virtual:pwa-register").then(({ registerSW }) => {
    const updateSW = registerSW({
      immediate: true,
      onNeedRefresh() {
        const sinceBoot = performance.now() - bootedAt
        if (sinceBoot <= BOOT_GRACE_MS) {
          // Fresh start. Activate the waiting SW; vite-plugin-pwa's
          // updateSW(true) handles skipWaiting + reload internally.
          void updateSW(true)
          return
        }

        // Mid-session: notify the user with a sticky toast. The action
        // button is the explicit user reload path. The background
        // reloadWhenIdle path is the safety net for users who ignore
        // or miss the toast.
        useNotificationStore.getState().addNotification({
          level: "info",
          title: "Neue Version verfügbar",
          message: "Klicke auf 'Jetzt neu laden', um zu aktualisieren.",
          duration: 0,
          action: {
            label: "Jetzt neu laden",
            onClick: () => {
              void updateSW(true)
            },
          },
        })
        reloadWhenIdle(() => window.location.reload())
      },
      onOfflineReady() {
        useNotificationStore.getState().addNotification({
          level: "success",
          title: "Offline ready",
          message: "Chatsune is now available offline.",
        })
      },
    })
  })
}
```

Notes:

- The `BOOT_GRACE_MS` constant is added at module scope, alongside the
  existing `SETTLE_MS` constant near the top of the file.
- `bootedAt` uses `performance.now()` — a monotonic clock unaffected
  by system clock changes. `Date.now()` would also work but is
  marginally less precise.
- `immediate: true` instructs vite-plugin-pwa to check for updates as
  soon as the SW is registered, rather than relying on the browser's
  default cadence. This shortens the boot-window race for users whose
  cached SW hasn't pre-fetched the new version.
- The mid-session toast uses `level: "info"` (purple, matches the
  existing offline-ready toast convention but distinct from
  success/warning/error). `duration: 0` triggers the sticky behaviour
  added in Task 1.
- The "Jetzt neu laden" button calls `updateSW(true)` directly — no
  `reloadWhenIdle` here. The user clicking the button is an explicit
  signal that they want to reload immediately, even mid-stream.
  Streaming-state loss is accepted; messages are persisted server-side.

**Boot-window auto-reload:**

**Mid-session toast + manual reload:**

7. Deploy build N+2.

   (typically up to ~60 s, depending on browser).

   and an action button "Jetzt neu laden". The toast does not
   auto-dismiss.
10. Click "Jetzt neu laden".
11. Expected: immediate page reload, app comes back as build N+2.
