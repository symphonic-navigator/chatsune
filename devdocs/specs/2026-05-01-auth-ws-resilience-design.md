# Auth and WebSocket Resilience

## Problem

Two related production issues, both rooted in how the frontend interprets
"the backend is gone" and "your session ended". Symptoms:

1. **The connection-lost screen never appears in deploy.** Stopping the
   backend container leaves the user on the chat view with no feedback.
   Locally it works, in deploy it doesn't.
2. **Sessions end "wild".** Users get logged out unexpectedly, see a
   blank chat view, and have to manually navigate to `/login`. No toast,
   no message.

### Root cause for issue 1: Traefik fallthrough

In the deploy (`fox.tidesson.net`), Traefik routes are:

```
chatsune-frontend  Host(...)                       priority=1   (catch-all)
chatsune-api       Host(...) && PathPrefix(/api)   priority=10
chatsune-ws        Host(...) && PathPrefix(/ws)    priority=10
```

When the backend container stops, Docker's Traefik provider drops the
backend services. The `/api` route has no target. Traefik falls through
to the only remaining matching route — the frontend catch-all. The
frontend container (nginx serving static files) receives `POST
/api/auth/kdf-params` and answers `405 Method Not Allowed`.

This is verified evidence from the live deploy: stopping the backend
produces 405 from the frontend, not 502 from Traefik. The frontend's
`api/client.ts` only flips `backendAvailable` to false on
`TypeError: Failed to fetch` (a real network error). A 405 with a
valid HTTP body counts as a successful response. The
`BackendUnavailablePage` (gated on `backendAvailable === false`) is
therefore never rendered.

Locally this is hidden because Vite's dev server has no fallthrough
target — a stopped backend produces a real connection-refused, which
does flip the flag.

### Root cause for issue 2: silent auth clearing

The frontend has four logout paths. Three of them do not navigate:

1. Logout button → `clear()` (no nav)
2. WS close 4001 + refresh fails → `clear()` (no nav)
3. HTTP 401 + refresh fails → `clear()` (no nav)
4. Account deletion → `clear()` (no nav)

The user remains on `/chat` after auth state flips. `AuthGuard` reacts
to route matches, not state changes, so nothing pushes the user to
`/login`. Combined with no toast or message, the experience is "the
chat just stopped working".

## Goal

The frontend never silently sits in a half-dead state. Two invariants:

- **If the backend is unreachable** (stopped, network gone, proxy
  fallthrough, anything), the user sees the connection-lost screen
  within a few seconds.
- **If the user's session ends** for any reason (token expired, refresh
  failed, must change password, account deleted), the user is taken to
  `/login` with a toast explaining what happened.

## Design

### 1. Backend marker header

A FastAPI middleware adds `X-Chatsune-Backend: 1` to every response on
the `/api/*` routes. The value is a stable marker; we keep it minimal
now and may extend it with a backend version string later (relevant
for Item 5, PWA update detection — out of scope here).

The middleware is registered once in `backend/main.py`. It runs
unconditionally on the `/api` mount and adds the header to every
response, including error responses (4xx and 5xx). Responses that
originate from the backend always carry the header; responses
generated by anything in front of or beside the backend (Traefik
fallthrough, frontend nginx, browser cache) do not.

The `/ws` routes are out of scope for the header — WebSocket upgrade
responses follow different semantics, and the WS pong timeout
(component 3) covers the connection-liveness signal.

### 2. Frontend health monitor (always-on)

The current `BackendUnavailablePage` component contains its own
`/api/health` poller. The poll is only active while the page is
mounted. We move polling out of the page and into a dedicated
service that runs for the lifetime of the app.

The service:

- Polls `GET /api/health` on a timer.
- Success criterion: HTTP `200` **and** the response carries the
  `X-Chatsune-Backend` header. Anything else is failure (405, 502,
  HTML body, missing header, JSON parse error, network error).
- On success: `eventStore.backendAvailable = true`, poll interval
  5 seconds.
- On failure: `eventStore.backendAvailable = false`, poll interval
  1 second (so the UI recovers quickly when the backend comes back).
- Suspended only when `document.hidden` (background tab); resumed on
  visibility change.

The `BackendUnavailablePage` itself becomes a pure presentation
component reading `backendAvailable` from the store; it no longer
contains poll logic.

### 3. API client: distinguish "backend authoritative" from not

The current API client treats any HTTP response as authoritative. We
add the header check:

- If a response carries `X-Chatsune-Backend`, the backend produced it
  and the status code is meaningful.
- If a response does not carry the header (Traefik fallthrough,
  frontend nginx, anything else), it is not authoritative for our
  protocol. We treat it as a backend-down signal:
  `backendAvailable = false` (the health monitor will confirm or
  recover), and the call fails with a `BackendUnavailableError`.

Concrete change in `core/api/client.ts`:

- `apiRequest` checks the response header before reading the body.
- Missing header → throw a typed error class
  `BackendUnavailableError`, set `backendAvailable: false`.
- Header present + 401 → existing refresh path runs. If the refresh
  itself returns a response with the header (legitimate `401`,
  meaning the refresh token is gone or invalid) → trigger
  `forceLogout('session_expired', ...)`. If the refresh call returns
  a response without the header (proxy fallthrough during refresh) →
  treat as backend-down, do NOT log out; the next refresh attempt
  after recovery may succeed.
- Header present + other error → bubble up as today.

This is the critical bit for issue 2: a 401 during a brief backend
hiccup must not silently log the user out. The header is the
distinguisher.

### 4. WebSocket pong timeout

The frontend already sends a ping every 30 seconds. We add a 10-second
pong timeout: after sending a ping, if no `pong` event arrives within
10 seconds, the socket is forcibly closed with code `1000` (normal
closure) and the existing reconnect logic engages.

`PONG_TIMEOUT_MS = 10_000` lives next to `PING_INTERVAL_MS` in
`connection.ts`. The pong handler clears the timer; the timer fires
`socket.close(1000)` if it elapses. Cleared on socket close /
reopen.

10 seconds is comfortably above any normal-network latency and tight
enough that a dead-but-not-closed socket is detected within one
ping cycle.

### 5. Centralised logout

Two functions live in a new module
`frontend/src/core/auth/logoutCoordinator.ts`:

```ts
// User pressed the logout button. Quiet, no toast, navigates to /login.
async function logout(): Promise<void>

// System-initiated. Navigates to /login, shows a toast with reason.
async function forceLogout(
  reason: 'session_expired' | 'must_change_password' | 'admin_revoked',
  userMessage: string,
): Promise<void>
```

Both call a shared internal `_doLogout()` that:

1. Calls `authApi.logout()` (best effort; failures are logged but do
   not block).
2. Disconnects the WebSocket.
3. Clears the auth store.
4. Navigates to `/login` via the React Router instance.

`forceLogout` additionally pushes a toast through the existing
`ToastContainer` mechanism with the user-facing message.

The four existing logout sites are updated:

- Logout button (in `useAuth.ts`) → `logout()`
- WS close 4001 + refresh fails → `forceLogout('session_expired',
  'Deine Sitzung ist abgelaufen. Bitte melde dich erneut an.')`
- WS close 4003 (must-change-password) → `forceLogout(
  'must_change_password', 'Bitte ändere dein Passwort.')`
- HTTP 401 + refresh fails (with header present, see component 3) →
  `forceLogout('session_expired', ...)`
- Account deletion → `logout()` (already user-initiated)

Manual messages stay German because that's the user-facing language
in this app. (Code identifiers stay British English per project
convention.)

### 6. UX behaviour

- `BackendUnavailablePage` renders the moment `backendAvailable` is
  `false`. No "wait for reconnect attempts to give up" intermediate
  state. The health monitor is the source of truth.
- The Topbar's connection-status pill (`reconnecting`, `connected`)
  remains as a quick at-a-glance indicator, but is no longer the
  only feedback in the failure case.
- Toasts on `/login` after `forceLogout` use the existing toast
  system. They auto-dismiss after the standard duration.

### Out of scope

- Refresh-token storage in localStorage (security downgrade,
  rejected).
- Token-rotation race in multi-tab (separate problem; not reported
  in production yet).
- Backend health endpoint enriched with database / Redis status —
  YAGNI.
- Traefik configuration changes (a backend fail-service that returns
  502 instead of falling through). Possible future improvement, not
  needed for the resilience guarantees above.
- PWA update detection (Item 5, separate spec).
- Pixel tablet PWA layout (Item 4, separate investigation).

## Manual verification

The product owner runs these against the deployed environment.

### Backend gone (issue 1)

1. Log in, start a chat session, voice if convenient.
2. `docker compose stop backend` on the VPS.
3. Within ~5 seconds, the `BackendUnavailablePage` appears. Network
   tab in DevTools shows health-poll requests, all failing
   (405 from frontend nginx).
4. `docker compose start backend`.
5. Within ~5 seconds, `BackendUnavailablePage` disappears, the app
   reappears, the WebSocket reconnects (Topbar pill goes back to
   green).

### Login during backend-down

6. With backend stopped, on the login page, click "Login".
7. The login form does not silently fail. The user sees a toast or
   error message. (Because `kdf-params` returns no header, the API
   client throws `BackendUnavailableError`, the login form catches
   it, the health monitor sets `backendAvailable: false`, the
   `BackendUnavailablePage` takes over.)

### Session expiring naturally

8. Log in normally.
9. On the VPS, manually delete the refresh token from Redis:
   `docker compose exec redis redis-cli` then `KEYS refresh:*` and
   `DEL <key>`.
10. Wait for the access token to expire (15 minutes), or trigger an
    early refresh by reloading the page after ~13 minutes.
11. The frontend attempts a refresh, the backend returns 401 with
    the header, `forceLogout('session_expired', ...)` runs.
12. The user is on `/login` with a toast saying "Deine Sitzung ist
    abgelaufen. Bitte melde dich erneut an."

### Backend hiccup during refresh (no spurious logout)

13. Log in, wait ~13 minutes so a token refresh is imminent.
14. `docker compose stop backend` for ~10 seconds, then start it.
15. During the gap, if the frontend tries to refresh, it should NOT
    log the user out. The `BackendUnavailablePage` may flash briefly,
    then disappear. The session continues.

### Manual logout (no regression)

16. Click the logout button.
17. The user is on `/login`, no toast.

### WS pong timeout (best-effort, optional)

18. With the backend running, use the browser's DevTools Network
    throttling to "Offline" mode for ~15 seconds while a session is
    active.
19. The WebSocket pill goes to `reconnecting` within ~10-30 seconds
    (one ping cycle + pong timeout).
20. Disable throttling. The pill returns to `connected`.

## Files affected

### Backend

- `backend/main.py` — register the response middleware.
- `backend/_middleware/backend_marker.py` (new) — middleware
  implementation.

### Frontend

- `frontend/src/core/api/client.ts` — header check in `apiRequest`,
  `BackendUnavailableError` class, refresh-failure routing.
- `frontend/src/core/health/healthMonitor.ts` (new) — always-on
  health-poll service.
- `frontend/src/core/auth/logoutCoordinator.ts` (new) — `logout()`
  and `forceLogout()`.
- `frontend/src/core/websocket/connection.ts` — pong timeout, route
  4001/4003 close handling through `forceLogout`.
- `frontend/src/core/hooks/useAuth.ts` — logout button calls
  `logout()` from the coordinator instead of the inline path.
- `frontend/src/app/pages/BackendUnavailablePage.tsx` — strip out
  poll logic, become pure presentation reading from the store.
- `frontend/src/App.tsx` — start the health monitor on mount; the
  BackendUnavailablePage gating already exists.
- `frontend/src/core/store/eventStore.ts` — no shape change; the
  `backendAvailable` flag already exists.
