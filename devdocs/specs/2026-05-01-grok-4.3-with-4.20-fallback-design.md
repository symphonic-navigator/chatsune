# Grok 4.3 (with Grok 4.20 voice fallback)

## Problem

The xAI integration currently exposes exactly one model — "Grok 4.1
Fast" — which toggles internally between
`grok-4-1-fast-reasoning` and `grok-4-1-fast-non-reasoning` based on
`request.reasoning_enabled`
(`backend/modules/llm/_adapters/_xai_http.py:267-275`). Two newer xAI
generations are not yet available to users:

- **Grok 4.20** — has both a reasoning slug
  (`grok-4.20-0309-reasoning`) and a non-reasoning slug
  (`grok-4.20-0309-non-reasoning`). Same toggle pattern as 4.1 Fast.
- **Grok 4.3** — reasoning-only. No non-reasoning slug exists.
  Voice/continuous-voice latency requirements rule out reasoning, so
  selecting Grok 4.3 in those modes would either error out or hang the
  user with multi-second latency. xAI may add a non-reasoning variant
  later but has not announced one.

The product owner reports a strong constituency on Discord requesting
4.3, considers the model qualitatively excellent, and wants both new
generations available — without forcing users to memorise xAI's model
matrix or making them think about which slug applies in which mode.

A second, smaller issue: the current "Grok 4.1 Fast" entry advertises
a 200 000-token context window. xAI has since lowered the
"affordable-tier" cut-off for that model to 128 000 tokens; beyond
128 k the price doubles. The advertised window should match the tier
the product is willing to pay for.

## Goal

Users can select **Grok 4.20** or **Grok 4.3** as ordinary entries in
the model browser, alongside Grok 4.1 Fast. Selecting Grok 4.3 works
in both reasoning-on and reasoning-off paths without errors, without
forcing users to understand that xAI has no non-reasoning 4.3 slug,
and without hiding that fact from users who *do* want to know. The
context window for Grok 4.1 Fast reflects xAI's 128 k affordable-tier
cap.

## Design

### Mechanism: per-model slug pair, with optional cross-generation fallback

Generalise the xAI adapter from a single hardcoded reasoning /
non-reasoning slug pair to a small mapping table keyed on `model_id`.
Each entry carries:

- `model_id` — the value Chatsune surfaces and stores (also what
  arrives back as `request.model`).
- `display_name`, `context_window`, billing/capability flags.
- `reasoning_slug` — the xAI API slug used when
  `request.reasoning_enabled` is true.
- `non_reasoning_slug` — the xAI API slug used when
  `request.reasoning_enabled` is false. **May be a slug from a
  different generation** when no same-generation variant exists.
- `remarks` — optional human-readable note (see "User-facing
  communication" below).

The Grok 4.3 entry uses `grok-4.20-0309-non-reasoning` as its
`non_reasoning_slug`. This is the workaround for xAI's missing 4.3
fast variant and the only entry where the two slugs cross
generations.

#### Mapping table

| `model_id`     | display_name   | context | reasoning slug              | non-reasoning slug              | remarks                                                       |
|----------------|----------------|---------|-----------------------------|---------------------------------|---------------------------------------------------------------|
| `grok-4.1-fast`| "Grok 4.1 Fast"| 128 000 | `grok-4-1-fast-reasoning`   | `grok-4-1-fast-non-reasoning`   | *(none)*                                                      |
| `grok-4.20`    | "Grok 4.20"    | 200 000 | `grok-4.20-0309-reasoning`  | `grok-4.20-0309-non-reasoning`  | *(none)*                                                      |
| `grok-4.3`     | "Grok 4.3"     | 200 000 | `grok-4.3`                  | `grok-4.20-0309-non-reasoning`  | "Falls back to Grok 4.20 (non-reasoning) when thinking is off." |

All three entries: `supports_reasoning=True`, `supports_vision=True`,
`supports_tool_calls=True`, `billing_category="pay_per_token"`.

The 200 000 / 128 000 numbers are the **policy** ceiling, not xAI's
upstream technical maximum. Beyond these tokens xAI's pricing changes
(doubles, per the product owner). The product is unwilling to surface
context windows the user cannot afford. "Compact and continue" for
endless chats is a separate, later piece of work.

### Adding `remarks` to `ModelMetaDto`

Extend `shared/dtos/llm.py::ModelMetaDto` and the matching
`frontend/src/core/types/llm.ts::ModelMetaDto` with one new optional
field:

```python
# shared/dtos/llm.py
remarks: str | None = None
```

```ts
// frontend/src/core/types/llm.ts
remarks?: string | null
```

Default `None` keeps pre-existing cached payloads readable without a
migration — same pattern already used for `is_deprecated`,
`billing_category`, `is_moderated` (see CLAUDE.md §Data-Model
Migrations and the existing field comments around
`shared/dtos/llm.py:25-41`).

This field is **adapter-set, read-only at the consumer**. Today only
the xAI adapter writes it; other adapters leave it `None`. Future
adapters may use it for any other vendor quirk.

A single string is the right shape: YAGNI vs. a list. If a future
model needs multiple remarks, the text can carry line breaks or
bullets, or the field migrates to a list at that point.

### xAI adapter changes

In `backend/modules/llm/_adapters/_xai_http.py`:

1. Replace the module-level `_XAI_MODEL_REASONING` /
   `_XAI_MODEL_NON_REASONING` constants with a typed mapping
   (e.g. an internal `_XaiModelEntry` dataclass + a dict keyed on
   `model_id`).
2. `XaiHttpAdapter.fetch_models` returns one `ModelMetaDto` per
   mapping entry instead of the single hardcoded one currently at
   line 346-358. The `remarks` field is set from the mapping.
3. `_build_chat_payload(request)` looks the entry up by
   `request.model`, then picks `reasoning_slug` or
   `non_reasoning_slug` based on `request.reasoning_enabled`. Today
   the function ignores `request.model` entirely (line 271-275); that
   is the bug we are fixing.
4. If `request.model` is unknown to the mapping (legacy persona
   referencing a removed slug, malformed input), fall back to the
   `grok-4.1-fast` entry and log a warning. This preserves the
   existing behaviour where any xAI completion request resolved to a
   working slug; no user is left with a hard error from a stale
   reference.

`x-grok-conv-id` cache hint forwarding (line 372-373) and all other
existing behaviour are unchanged.

### Frontend: third-line `remarks` rendering

In `frontend/src/app/components/model-browser/ModelBrowser.tsx`,
inside `ModelRow` (the row layout starts at line 338), append a third
information line to the existing two-line column. The current rows
look like:

```
★ Grok 4.3                                            [R] [V] [T]
  grok-4.3 · ctx 200 000
```

They become, when `remarks` is non-empty:

```
★ Grok 4.3                                            [R] [V] [T]
  grok-4.3 · ctx 200 000
  Falls back to Grok 4.20 (non-reasoning) when thinking is off.
```

Concretely: after the existing meta-line `<div>` that ends around
line 393, render a sibling `<div>` only when `model.remarks` is
truthy. Same column container, smaller/dimmer than the meta line so
it does not compete with the model id. Suggested classes:
`mt-0.5 text-[11px] text-white/40 italic` — close in weight to the
meta line, but dimmed and italicised so it reads as commentary.
Wraps freely; no truncation.

When `model.remarks` is empty/null (every other model today), the
sibling `<div>` is not rendered — no whitespace reserved, layout
unchanged.

### What is *not* shown to the user

There is no live indicator at request time that the Voice path is
swapping in 4.20 for a 4.3 request. The product position is that the
user does not need to think about it on a per-message basis; the
remarks line at model-selection time is sufficient disclosure (and is
significantly more disclosure than e.g. OpenAI gave users for years
when their voice path used a smaller model).

For backend debugging, the existing `LLM_TRACE` line at
`_xai_http.py:378-383` already logs the outgoing payload — including
the resolved API slug. No additional logging is added.

### Backwards compatibility

- Existing personas referencing `xai:grok-4.1-fast` continue to work,
  and silently get the new 128 000 context window. No migration.
- Existing personas referencing the previous (Phase-1-only) xAI
  unique IDs `xai:grok-4` or `xai:grok-3`: unchanged from today —
  these are already not in the model list. The unknown-model
  fallback in step 4 above keeps requests routable.
- Older frontend caches lacking `remarks` continue to work; the
  field is optional.

## Out of scope

- "Compact and continue" for endless chats — a separate concern. The
  context-window caps in this spec are the policy ceiling regardless
  of whether compact-and-continue ever ships.
- Per-user override of `remarks` (e.g. "Hide this hint"). YAGNI.
- A live UI hint inside the chat composer or the voice indicator
  surfacing the swap on a per-message basis. Decided against; the
  remarks line is the single point of disclosure.
- Adding a `requires_reasoning` capability flag on `ModelMetaDto`.
  Grok 4.3 reports `supports_reasoning=True` (which is true) and the
  adapter's slug-pair mechanism handles the missing variant
  internally. A "must always reason" flag would have no consumer
  today — YAGNI.
- Other adapters' use of `remarks`. They simply leave it `None`. We
  do not retro-fit remarks onto, e.g., Mistral or nano-gpt models in
  this change.
- Changes to `ModelConfigModal` (per-user custom display name /
  context / reasoning override). Users can still override the
  context window of any of these three models if they want to; the
  validator floor of 80 000 in `shared/dtos/llm.py:90` remains in
  place.
- Image generation. xAI's image group definition
  (`_xai_image_groups.py`) is unaffected.

## Files affected

- `shared/dtos/llm.py` — add `remarks` field to `ModelMetaDto`.
- `frontend/src/core/types/llm.ts` — add `remarks?` to
  `ModelMetaDto`.
- `backend/modules/llm/_adapters/_xai_http.py` — replace constants
  with mapping table, expand `fetch_models`, route
  `_build_chat_payload` through the mapping, add unknown-model
  fallback.
- `frontend/src/app/components/model-browser/ModelBrowser.tsx` —
  conditional third-line render in `ModelRow`.

No new files. No new dependencies. No store-shape changes. No
backend module-boundary changes.

## Manual verification

The product owner runs these on the deployed environment, with an
xAI connection already configured.

### Model browser shows the three xAI entries

1. Open the model browser.
2. **Expected:** under the xAI connection, three entries are listed:
   "Grok 4.1 Fast" (ctx 128 000), "Grok 4.20" (ctx 200 000), "Grok
   4.3" (ctx 200 000). Each shows `[R] [V] [T]` capability badges.
3. **Expected:** the "Grok 4.3" row has a third information line
   reading roughly "Falls back to Grok 4.20 (non-reasoning) when
   thinking is off." The other two xAI rows do **not** have a third
   line. Other adapters' rows are visually unchanged.
4. **Expected:** the "Grok 4.1 Fast" row reads `ctx 128 000`, not
   `ctx 200 000` (regression on the affordable-tier cap).

### Grok 4.20: reasoning toggle works

5. Pick a persona, set its model to Grok 4.20, reasoning on. Send a
   text message.
6. **Expected:** the conversation completes normally. Backend
   `LLM_TRACE` line shows the outgoing payload with
   `"model": "grok-4.20-0309-reasoning"`.
7. Toggle the persona's reasoning off, send another message.
8. **Expected:** completes normally. Backend trace shows
   `"model": "grok-4.20-0309-non-reasoning"`.

### Grok 4.3: reasoning on uses 4.3 slug

9. Set the persona's model to Grok 4.3, reasoning on. Send a text
   message.
10. **Expected:** completes normally, possibly slower than 4.20 due
    to thinking. Backend trace shows `"model": "grok-4.3"`.

### Grok 4.3: voice / non-reasoning falls back to 4.20

11. Same persona, switch to continuous voice mode (or any path that
    sets `reasoning_enabled=false`). Speak a short prompt.
12. **Expected:** the assistant responds with low voice-mode
    latency. Backend trace shows
    `"model": "grok-4.20-0309-non-reasoning"`. The voice indicator
    does **not** display anything model-specific about the swap.
13. Quit voice. Send a text message with reasoning still off.
14. **Expected:** completes normally. Trace again shows the 4.20
    non-reasoning slug.

### Existing 4.1 Fast personas keep working

15. If any saved persona still uses `xai:grok-4.1-fast`, send a
    message with reasoning on, then off.
16. **Expected:** both succeed. Trace shows `grok-4-1-fast-reasoning`
    and `grok-4-1-fast-non-reasoning` respectively. The persona's
    advertised context window in the model picker now reads
    128 000 (it previously read 200 000); send a long-context
    message to confirm nothing is silently truncated below the new
    cap.

### Unknown model fallback (regression safety)

17. Manually edit a persona's model field in MongoDB to a stale
    xAI value such as `xai:grok-4`. Send a message.
18. **Expected:** the request resolves through the fallback to
    `grok-4-1-fast-*` rather than 5xx-ing. Backend logs a warning
    naming the unknown model_id.

If any scenario fails, do NOT merge; report which step and what
was observed.
